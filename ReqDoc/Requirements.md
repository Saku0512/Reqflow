## 1. プロジェクト概要

### 1.1 背景

ソフトウェア開発チームにおいて、PdM（プロダクトマネージャー）が要件定義書を作成してからエンジニアへタスクを指示するまでのプロセスは、現状ツールが分断されており手動での二重管理が発生している。要件定義書はNotionやConfluenceで管理し、タスクはJiraやGitHub Projectsで別途起票するという運用が一般的であり、要件とタスクのトレーサビリティが失われやすい。

### 1.2 目的

要件定義書の作成・管理から、タスクへの変換・GitHub Projects連携・進捗管理までを一気通貫で行えるデスクトップアプリケーション「Reqflow」を開発し、PdMのワークフロー効率化とチームの認識齟齬を解消する。

### 1.3 スコープ

**対象とする範囲：**

- 要件定義書の作成・編集・管理
- タスクの進捗管理・担当者管理（スタンドアロン動作）
- GitHub IssueおよびGitHub Projectsへのタスク連携（オプション機能）
- AIによる要件定義書ドラフト自動生成（v1.0）

**対象外とする範囲：**

- Webブラウザからのアクセス（デスクトップ専用）
- GitHub以外のプロジェクト管理ツール連携（Jira・Linear等）
- チームメンバーへのリアルタイム通知機能
- 課金・ライセンス管理機能

## 2. 現状分析（As-Is）

### 2.1 現状の業務フロー

```
PdM がヒアリング
    ↓
Notion / Confluence 等で要件定義書を手書き
    ↓
要件定義書を読んでPdMが手動でタスクを起票（GitHub Issues / Jira）
    ↓
TL / PL がタスクをエンジニアにアサイン
    ↓
進捗確認のたびに要件定義書とタスクを行き来して照合
```

### 2.2 現状の課題

| # | 課題 | 影響 |
|---|------|------|
| 1 | 要件定義書とタスクが別ツールで管理されており、トレーサビリティがない | 「この実装はどの要件から来たか」が追えない |
| 2 | 要件からタスクへの変換が手動のため抜け漏れが発生する | 未実装の要件が発生する |
| 3 | 要件定義書の承認フローが属人的・非定型 | ステータスが曖昧なまま開発が進む |
| 4 | 未決定事項の管理が要件定義書の中で埋もれる | 期限切れの未決定事項が放置される |

## 3. 目標・ゴール（To-Be）

### 3.1 理想の業務フロー

```
PdM が Reqflow 上で要件定義書を作成
    ↓
機能要件の各項目からワンクリックでGitHub Issueを生成
    ↓
GitHub Projects と自動同期し、TL / PL がアサイン
    ↓
Reqflow のタスクボードで進捗確認（要件へのリンクが常に保持される）
```

### 3.2 達成目標

| 目標 | 指標 |
|------|------|
| 要件→タスク変換の工数削減 | 手動起票ゼロ（自動生成率100%） |
| トレーサビリティの確保 | 全タスクに要件IDが紐付いた状態を維持 |
| 未決定事項の管理 | 期限切れ未決定事項をゼロに近づける |

## 4. ステークホルダー定義

| 役割 | 説明 | 本アプリにおける立ち位置 |
|------|------|--------------------------|
| PdM（プロダクトマネージャー） | プロダクトの価値・売上を最大化することに集中する。スクラムのPO（プロダクトオーナー）に近い役割 | **主ユーザー**。要件定義書の作成・承認・タスク生成を行う |
| PM（プロジェクトマネージャー） | プロジェクト計画の遂行と外部（クライアント・他部署・経営層等）との調整を主に担う | 計画進捗・全体スケジュールの把握のためにダッシュボードを参照する。Reqflowの直接操作は少ない |
| TL（テックリード） | 技術的な意思決定・アーキテクチャ設計を担う | 機能要件の技術的妥当性をレビューし、タスクをエンジニアにアサインする |
| PL（プロジェクトリーダー） | チームごとに1名存在。PMが外部調整を担う一方、チーム内部の状態管理を担う。チームの課題・進捗をPMに報告する。1プロジェクトに複数チームがある場合、PLも複数人になる | チーム内のタスク進捗・ボトルネックをタスクボードで管理し、PMへ状況を伝える橋渡し役 |
| Full-cycleエンジニア | 設計〜実装〜テストまでを一貫して担当する | TL・PLからアサインされたタスクを受け取り実装する。必要に応じて要件定義書を参照する |
| 開発者（本アプリの） | Saku0512 | 設計・開発・保守を担当 |

## 5. 業務要件

### 5.1 プロジェクト管理業務

PdMは複数のプロジェクトを並行して管理できる必要がある。各プロジェクトは独立した要件定義書とタスクボードを持つ。

### 5.2 要件定義書作成業務

PdMは以下の構成に沿って要件定義書を作成できる必要がある。

- 業務要件（課題・背景・業務フロー）
- 機能要件（機能一覧・画面一覧・バッチ処理一覧）
- 非機能要件（性能・セキュリティ・可用性等）
- 制約条件

各セクションはMarkdown形式で記述し、編集内容は自動保存される。

### 5.3 未決定事項管理業務

要件定義書の作成中に判明した未決定事項を、期限付きでインライン登録・管理できる必要がある。期限が近い・超過した事項はダッシュボードで警告表示する。

### 5.4 タスク生成業務

機能要件の各項目からGitHub Issueを自動生成し、GitHub Projectsの指定ボードに追加できる必要がある。生成されたIssueには要件IDが自動でタグ付けされる。

### 5.5 進捗管理業務

GitHub Projectsと同期し、タスクの担当者・ステータス・期限をReqflow内で確認・更新できる必要がある。

## 7. 非機能要件

IPAの「非機能要求グレード2018」に基づき定義する。

### 7.1 可用性

| 項目 | 要件 |
|------|------|
| 動作環境 | macOS 13以上、Windows 10以上 |
| クラッシュ時の挙動 | 自動保存により編集内容を保全する（データロスなし） |
| オフライン動作 | 要件定義書の作成・編集はオフラインで動作する（GitHub連携機能を除く） |

### 7.2 性能・拡張性

| 項目 | 要件 |
|------|------|
| 起動時間 | 3秒以内 |
| 自動保存のレスポンス | 編集停止から1秒以内に保存完了 |
| 対応プロジェクト数 | 100プロジェクト以上を想定 |
| 要件セクション数 | 1プロジェクトあたり50セクション以上 |

### 7.3 セキュリティ

| 項目 | 要件 |
|------|------|
| GitHub PAT の保管 | OS のKeychain / Credential Manager に暗号化して保存する（平文保存禁止） |
| ローカルDBの保護 | SQLiteファイルへの直接アクセスはアプリ経由のみとする |
| 通信 | GitHub API との通信はHTTPS必須 |

### 7.4 運用・保守性

| 項目 | 要件 |
|------|------|
| DBマイグレーション | スキーマ変更はマイグレーションファイルで管理する |
| ログ | エラーログをローカルファイルに出力する |
| アップデート | Wailsのビルド成果物を差し替えることでアップデートできる |

### 7.5 移行性

初期リリースのため、既存データからの移行要件はなし。将来的にNotionやConfluenceからのインポート機能を検討する（v1.0以降）。

## 8. 制約条件

| # | 制約 | 理由 |
|---|------|------|
| C-01 | フロントエンドはSvelteを使用する | 開発者のスキルセット・Wailsとの親和性 |
| C-02 | バックエンドはGoで実装する | 開発者のスキルセット・型安全性 |
| C-03 | デスクトップフレームワークはWails v2を使用する | GoをファーストクラスでサポートするTauriの代替 |
| C-04 | データベースはSQLiteを使用する | サーバーレス・デスクトップアプリに適した軽量DB |
| C-05 | GitHub連携はGraphQL API v4を使用する | GitHub Projectsの操作にはGraphQLが必要 |
| C-06 | GitHub連携はオプション機能とし、未設定でも全中核機能が動作する | リポジトリアクセス権が限られる環境への対応 |
| C-07 | Webブラウザからのアクセスは提供しない | デスクトップアプリとして設計するため |

## 10. データ要件

### 10.1 主要エンティティ

```
projects
    ├── requirement_docs（1:1）
    │       ├── requirement_sections（1:N）
    │       └── open_issues（1:N）
    └── tasks（1:N）※ GitHub Issueと同期
```

### 10.2 データスキーマ

```sql
-- プロジェクト
CREATE TABLE projects (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    description TEXT,
    status      TEXT DEFAULT 'draft',
    github_repo TEXT,          -- owner/repo 形式
    github_project_id TEXT,    -- GitHub Projects ID
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 要件定義書
CREATE TABLE requirement_docs (
    id          TEXT PRIMARY KEY,
    project_id  TEXT NOT NULL REFERENCES projects(id),
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 要件セクション
CREATE TABLE requirement_sections (
    id          TEXT PRIMARY KEY,
    doc_id      TEXT NOT NULL REFERENCES requirement_docs(id),
    type        TEXT NOT NULL,  -- 'business' | 'functional' | 'non_functional' | 'constraint'
    title       TEXT NOT NULL,
    content     TEXT,           -- Markdown
    order_index INTEGER DEFAULT 0,
    status      TEXT DEFAULT 'draft'  -- 'draft' | 'review' | 'approved'
);

-- 未決定事項
CREATE TABLE open_issues (
    id          TEXT PRIMARY KEY,
    doc_id      TEXT NOT NULL REFERENCES requirement_docs(id),
    description TEXT NOT NULL,
    due_date    DATE,
    resolved    BOOLEAN DEFAULT FALSE,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- タスク（GitHub Issueと同期）
CREATE TABLE tasks (
    id              TEXT PRIMARY KEY,
    project_id      TEXT NOT NULL REFERENCES projects(id),
    section_id      TEXT REFERENCES requirement_sections(id),  -- 要件へのトレーサビリティ
    github_issue_id TEXT,
    title           TEXT NOT NULL,
    status          TEXT DEFAULT 'todo',  -- 'todo' | 'in_progress' | 'done'
    assignee        TEXT,
    due_date        DATE,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 10.3 データ保持方針

| データ種別 | 保持場所 | 保持期間 |
|-----------|----------|----------|
| プロジェクト・要件定義書 | ローカルSQLite | 削除操作まで永続 |
| GitHub PAT | OS Keychain | ユーザーが削除するまで |
| エラーログ | ローカルファイル | 30日でローテーション |
