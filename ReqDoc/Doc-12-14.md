## 1. 業務機能構成表（DD.5.2）

Reqflowが支援する業務を階層で整理する。

```
Reqflow支援業務
│
├── 1. プロジェクト管理業務
│   ├── 1.1 プロジェクト作成
│   ├── 1.2 プロジェクト一覧確認
│   └── 1.3 プロジェクト設定変更
│
├── 2. 要件定義書作成・管理業務
│   ├── 2.1 要件定義書の新規作成（テンプレート選択）
│   ├── 2.2 セクションの編集（Markdown）
│   ├── 2.3 セクションのステータス管理（Draft→Review→Approved）
│   ├── 2.4 セクションの追加・並び替え・削除
│   └── 2.5 未決定事項の管理
│       ├── 2.5.1 未決定事項の登録（期限付き）
│       └── 2.5.2 未決定事項の解決済みマーク
│
├── 3. タスク管理業務
│   ├── 3.1 タスクの作成（要件からの生成 or 手動作成）
│   ├── 3.2 タスクへの担当者アサイン
│   ├── 3.3 タスクのステータス更新（todo→in_progress→done）
│   └── 3.4 進捗確認（ダッシュボード・カンバンボード）
│
└── 4. GitHub連携業務（オプション）
    ├── 4.1 GitHub PAT設定・認証
    ├── 4.2 要件からGitHub Issue自動生成
    └── 4.3 GitHub Projectsとのステータス同期
```

## 2. 業務フロー As-Is（現状）（DD.5.3）

### メインフロー：要件定義〜タスク管理

```mermaid
sequenceDiagram
    autonumber
    actor PdM
    participant External as 外部ツール<br/>(Notion/Slack等)
    participant GitHub as GitHub Issues
    actor Team as 開発チーム<br/>(TL/PL/Eng)

    Note over PdM, External: 1. 情報収集・ドキュメント作成
    PdM->>External: ヒアリング・議事録の確認
    PdM->>External: 要件定義書を「手書き」作成

    Note over PdM, GitHub: 2. 手動でのタスク展開
    PdM->>GitHub: 要件を読みながら手動でIssue起票
    Note right of PdM: ※転記ミス・抜け漏れリスク

    Note over PdM, Team: 3. 情報共有
    PdM->>Team: 口頭 or Slackで「要件書」と「Issue」を共有

    loop ステータス確認のたびに発生
        Team-->>GitHub: 実装・進捗更新
        PdM->>GitHub: 進捗をチェック
        PdM->>External: 要件定義書のステータスを手動更新
        Note over PdM, External: NotionとGitHubを何度も往復
    end
```

**課題ポイント：**
- 要件定義書とIssueが常に乖離するリスクがある
- 進捗確認のたびに複数ツールを往復する
- 未決定事項が埋もれ、放置されやすい

## 3. システム化業務フロー To-Be（Reqflow導入後）（DD.5.4）

### メインフロー（GitHub連携あり）

```mermaid
sequenceDiagram
    autonumber
    actor PdM
    participant RF as Reqflow (本システム)
    participant GH as GitHub
    actor Team as チーム (TL/PL/Eng)

    Note over PdM, RF: 1. プロジェクト基盤の構築
    PdM->>RF: プロジェクト作成
    PdM->>RF: 要件定義（セクション単位で編集）
    Note right of RF: オートセーブ & バージョン管理

    Note over PdM, RF: 2. 懸案事項の管理
    PdM->>RF: 未決定事項のインライン登録
    Note right of RF: 期限アラート機能

    Note over PdM, GH: 3. タスク展開の自動化
    PdM->>RF: ワンクリックIssue生成
    RF->>GH: Issue自動生成（要件IDをラベル付与）
    Note over RF, GH: トレーサビリティの確立

    Note over RF, GH: 4. リアルタイム同期
    GH-->>RF: ステータス更新 (Close等)
    Note right of RF: 要件側の進捗に自動反映

    Note over PdM, Team: 5. 状況可視化と共有
    RF-->>PdM: ダッシュボードで進捗・承認状況を確認
    RF->>Team: タスクボード共有
```

### サブフロー（GitHub連携なし・スタンドアロン）

```mermaid
sequenceDiagram
    autonumber
    actor PdM
    participant RF as Reqflow (本システム)
    actor Team as チーム (TL/PL/Eng)

    Note over PdM, RF: 1. プロジェクト・要件作成
    PdM->>RF: プロジェクト作成
    PdM->>RF: 要件定義書の編集（セクション管理）

    Note over PdM, RF: 2. 内部タスク展開
    PdM->>RF: 手動 または ワンクリックでタスク作成
    Note right of RF: 要件IDとタスクを内部で強固に紐付け<br/>(GitHub連携不要)

    Note over RF, Team: 3. チーム内進捗管理
    RF-->>Team: タスクボード（カンバン等）の共有
    Team->>RF: 進捗ステータスを更新 (todo -> done)

    Note over PdM, RF: 4. 状況の可視化
    RF-->>PdM: ダッシュボードで進捗・要件充足率を確認
    Note right of RF: 要件単位での「完了」を自動判定
```

**To-Beのポイント：**
- GitHub連携の有無にかかわらず、要件とタスクは常に紐付いている
- 進捗確認はReqflowのダッシュボード1画面で完結する
- 未決定事項は期限アラートで見逃しを防止する
